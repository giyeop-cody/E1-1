## 1) docker 포트 매핑 증거

`docker run -d -p 8080:80 이미지:태그`

포트 매핑을 하는 이유?
도커 컨테이너는 docker network bridge에 의해서 격리된 네트워크 네임스페이스를 갖음.
호스트와는 다른 가상IP를 가지게되고 이 때 외부에서 컨테이너 내부로 접속하기 위한 경로가 없음. 그렇기에 외부에서 접근하려면 호스트의 특정한 포트로 들어와야하고 이를 컨테이너 내부 포트로 전달헤줘야함. 이를 위해서 컨테이너는 고정된 포트가 필요하며 호스트가 어떤 포트가 어떤 컨테이너와 연결되었는지 알아야 하기 때문에 포트 매핑을 해줌.

### NAT 네트워크 어드레스 트랜스레이트
공유기나 호스트 OS에서 격리된 하위 네트워크로 패킷을 보내주는 시스템. 다이나믹 포트 또는 일회성 포트를 통해서 내부 망의 단말이 네트워크와 통신할 수 있음. 여기서 역으로 외부에서 내부망의 단말에 접근하려면 고정된 포트가 필요하고 이걸 정해서 연결해주는 것이 포트 포워딩 ex) 80번 포트는 A주소를 가진 단말기에만 보내겠다

### NPAT 네트워크 어드레스 포트 트랜스레이트
주소 뿐만 아니라 포트 변환하는 시스템으로 외부에서 접근할 포트와 내부에서 쓰는 포트를 1:1 매핑해줌. ex) 8080포트로 들어온 요청은 A주소를 가진 단말에게 80포트로 보내겠다.

#### 포트 매핑
포트만 매핑해줌. ex) 8080포트로 들어오면 A컨테이너에 80으로 전달

## 3) 포트 매핑 제현하기

`docker run -p hostport:containerport image`는 매번 사용하는 것이 번거로운

해결법 1 dockerfile에 하기
```FROM nginx:latest
# 실제 포트를 여는 게 아니라, 80번을 쓴다는 '메모'입니다.
EXPOSE 80
```

강제 불가능함.

해결법 2 docker-compose.yml에 하기
```version: '1'
services:
  web:
    build: .
    ports:
      - "8080:80"  # 여기서 포트 매핑을 고정할 수 있습니다.
```

강제함.

## 4) 만일 포트가 이미 사용 중일 경우 트러블슈팅
확인: netstat -ano 또는 lsof -i :[포트번호] 명령어를 통해 해당 포트를 점유 중인 프로세스 ID(PID)를 확인.

판단: 해당 프로세스가 필수 서비스인지, 아니면 종료해도 되는 좀비 프로세스인지 판단.

조치: * 기존 프로세스 종료 (kill -9 [PID]) 후 재실행. <- 필요하면 살리고 아니면 죽인다.

또는, docker-compose.yml에서 호스트 쪽 포트 번호를 다른 번호로 변경(예: 8080:80 → 8081:80).

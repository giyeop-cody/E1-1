# Docker 볼륨 영속성 검증

## 1) Docker 볼륨 생성
`docker volume create` 명령어로 docker volume을 생성
![volume create](../img/스크린샷%202026-04-04%20오후%208.50.12.png)

## 2) Docker 볼륨 연결

### 가) Docker 볼륨 파일을 연결
`docker run -v 볼륨_경로:컨테이너_내_마운트_경로` 명령어를 통해서 컨테이너를 실행할 때 볼륨을 해당 경로로 연결
![검증](../img/스크린샷%202026-04-04%20오후%208.56.11.png)

### 나) 호스트의 디렉토리를 볼륨으로 바인드 마운트
`docker run -v 호스트_디렉토리_경로:컨테이너_내_마운트_경로`
![검증](../img/스크린샷%202026-04-06%20오후%2010.25.28.png)
바운드 마운트은 도커 볼륨을 만들어 컨테이너에 연결하는 것이 아닌 호스트의 절대 경로 디렉토리를 볼륨으로 마운트하는 것.

바운드 마운트된 호스트 디렉토리에서 파일 변경 시 반영 확인

docker volume은 바인드 마운트와 달리 도커에서 직접관리를 하기에 호스트의 의존성이 낮고 어떤 컨테이너에서도 쉽게 볼륨을 마운트하여 사용하기 쉽다는 이점이 있습니다.

|  | docker 볼륨 | 바인드 마운트 |
| :---: | :---: | :---: |
| 관리주체 | docker | 호스트 |
| 격리성 | 호스트와 격리됨 | 호스트에서 직접 수정 가능 |
| 이식성 | 높음 | 낮음(호스트에 종속적: 경로, 파일 시스템 등) |

## 3) Docker 컨테이너 삭제 전후 볼륨 데이터 영속성 검증
`docker ps`로 실행 중인 컨테이너를 확인

`docker stop 컨테이너_id`로 해당 컨테이너를 종료

`docker container rm -f 컨테이너_id`로 해당 컨테이너를 삭제

`docker run -d -v 볼륨_경로:컨테이너_내_마운트_경로 docker_이미지`로 신규 컨테이너 생성 및 실행

`docker exec -it 컨테이너_id sh`로 내부 터미널로 이동하여 쉘을 실행

`cd 컨테이너_내_마운트_경로`로 이동 후 `ls`와 `cat`를 이용해서 기존 파일이 존재 여부와 내용 검증

![검증](../img/스크린샷%202026-04-04%20오후%208.56.11.png)

볼륨은 컨테이너가 삭제되도 그 데이터가 유지(데이터의 영속)되기 때문에 컨테이너와 볼륨을 독립적으로 되어 있고 서로 데이터와 실행 환경이 서로 격리되어 있다고 생각합니다. 이것으로 비정상 종료를 대비하거나 컨테이너 삭제 이후에도 데이터를 보전할 수 있는 이점이 생기고 볼륨을 통째로 백업해두는 것으로 데이터의 복원할 수 있는 이점도 생깁니다.
